### 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | Gradio 5.x (`gr.Blocks`) | 5 个 Tab 页，SSE 流式渲染 |
| 后端 | FastAPI + Uvicorn | RESTful API，Port 8000 |
| 通信协议 | HTTP/1.1 + SSE (`text/event-stream`) | 流式问答使用 SSE |
| 向量库 | Milvus Lite (文件级) | 单进程访问，所以前端不直连 |
| 嵌入模型 | BAAI/bge-m3 (sentence_transformers) | Dense 1024 维 + Sparse |
| 重排序 | BAAI/bge-reranker-v2-m3 (FlagEmbedding) | Cross-encoder 精排 |
| LLM | Ollama (qwen3.5:4b) + 阿里云百炼 | Provider 模式 |
| 数据存储 | SQLite (file_metadata.db) | 文件注册表 |

### 进程架构

```
┌─────────────┐     HTTP/SSE      ┌──────────────┐
│  Gradio UI  │ ◄──────────────► │  FastAPI      │
│  Port 7860  │                   │  Port 8000    │
│  (客户端)    │                   │  (所有数据操作) │
└─────────────┘                   └──────────────┘
                                         │
```

**关键设计**: Gradio UI 不直接访问任何数据库或向量库。所有数据操作通过 HTTP 调用 FastAPI。这是因为 Milvus Lite 仅支持单进程访问。

```python
API_BASE = "http://localhost:8000"

def _api(method: str, path: str, **kwargs) -> dict | list | str:
    """通用 API 请求封装，所有前端 API 调用都经过此函数"""
    url = f"{API_BASE}{path}"
    timeout = kwargs.pop("timeout", 120)  # 默认 120s
    resp = requests.request(method, url, timeout=timeout, **kwargs)
    if resp.status_code >= 400:
        return f"[ERROR] {resp.status_code}: ..."
    return resp.json()
```

### 流式问答的 UI 渲染逻辑 (关键代码路径)

```python
def chat_stream(query, history, conv_id, domain_filter, top_k):
    """SSE 流式聊天 — token 级实时渲染"""
    # 1. 创建会话 (如无)
    if not conv_id:
        conv_id = _create_conversation()

    # 2. 添加用户消息 + 占位消息
    history.append({"role": "user", "content": query})
    history.append({"role": "assistant", "content": "🔍 检索中..."})

    # 3. 发起 SSE 流式请求 (timeout: 120s 连接, 900s 读取)
    resp = requests.post(f"{API_BASE}/ask/stream",
        json={"query": query, "top_k": top_k, ...},
        stream=True, timeout=(120, 900))

    # 4. 逐行解析 SSE 事件
    for line in resp.iter_lines():
        data = json.loads(line[6:])  # 去掉 "data: " 前缀

        if data["status"] == "searching":
            history[-1] = "🔍 检索中..."       # 心跳1
        elif data["status"] == "thinking":
            history[-1] = "💭 思考中..."       # 心跳2 (0.3s 动画)
        elif data.get("token"):
            full_answer += data["token"]
            # 限流: 最多每 80ms 刷新一次 UI
            if now - last_yield > 0.08:
                yield history, conv_id         # 实时更新界面
        elif data["done"]:
            break  # 最终显示完整答案
```

## 4. 核心数据流: `/ask/stream` 端到端

这是系统最关键的交互路径。一次用户提问经历 **7 个阶段**:

```
用户输入: "10kV配电线路的最小安全距离是多少？"
  │
  ▼
┌─────────────────────────────────────────────────────────────────┐
│ 阶段 1: Gradio UI → POST /ask/stream                            │
│                                                                  │
│ Request Body:                                                    │
│ {                                                                │
│   "query": "10kV配电线路的最小安全距离是多少？",                    │
│   "top_k": 15,                                                   │
│   "domain_filter": null,        // null = 自动识别                │
│   "conversation_id": "uuid-xxx" // null = 单轮问答               │
│ }                                                                │
│                                                                  │
│ Headers:                                                         │
│   Content-Type: application/json                                 │
│   Accept: text/event-stream                                      │
└──────────────────────┬──────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────────┐
│ 阶段 2: FastAPI 端点 → Retriever.search()                        │
│                                                                  │
│ 返回第一个 SSE 心跳:                                              │
│   data: {"status": "searching", "done": false}                   │
│                                                                  │
│ ┌─ Stage 0: QueryAnalyzer.analyze(query) ────────────────────┐  │
│ │  a. 提取文档编号 (正则匹配 GB/DL/Q/GDW 等模式)               │  │
│ │  b. jieba 分词 → 逐词匹配 domain_keywords                   │  │
│ │  c. 消歧规则运行 (变电/配电冲突, 送电输电/配电冲突)           │  │
│ │  d. 提取电压等级/专业/设备/发布级别/年份                     │  │
│ │  e. 同义词扩展 (消防→防火,灭火; 接地→接地装置等)             │  │
│ │  f. 构建 Milvus filter 表达式                                │  │
│ │                                                              │  │
│ │  输出: AnalyzedQuery {                                       │  │
│ │    query_type: "specification_lookup",                       │  │
│ │    domain: "配电",                                            │  │
│ │    voltage_level: "10kV",                                     │  │
│ │    expanded_terms: ["安全距离", "最小间距", "净距"],          │  │
│ │    filter_expr: "domain == '配电' && voltage_level == '10kV'" │  │
│ │  }                                                           │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                                                                  │
│ ┌─ Stage 1: 粗召回 (Hybrid Search) ─────────────────────────┐  │
│ │  a. Embedder.encode_query(query)                            │  │
│ │     → dense_vector (1024-dim) + sparse_vector (BGE-M3)      │  │
│ │  b. MilvusStore.hybrid_search()                             │  │
│ │     - dense_req: COSINE 相似度, limit=50                    │  │
│ │     - sparse_req: IP 内积, limit=50                         │  │
│ │     - ranker: WeightedRanker(0.7, 0.3)  ← 加权融合          │  │
│ │     - filter: domain == '配电' && voltage_level == '10kV'   │  │
│ │     → 返回 top-50 候选                                       │  │
│ │                                                              │  │
│ │  输出: 50 条 coarse candidates                               │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                                                                  │
│ ┌─ Stage 2: 精排 (Reranking) ───────────────────────────────┐  │
│ │  a. Reranker.rerank(query, candidates, top_k=15)            │  │
│ │     - BGE-Reranker cross-encoder 逐对打分 (语义相关性)       │  │
│ │     - metadata_boost: 加法归一化 (上限 1.15x)                │  │
│ │       · 文档编号精确匹配: +0.05                              │  │
│ │       · 文件名匹配: +0.10                                    │  │
│ │       · 域匹配: +0.05                                        │  │
│ │       · 电压匹配: +0.05                                      │  │
│ │       · 图纸查询降权: -0.10                                  │  │
│ │     → 最终得分 = semantic_score * min(1.15, 1.0 + boost)    │  │
│ │                                                              │  │
│ │  输出: top-15 精排结果                                        │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                                                                  │
│ 返回 SearchResponse { query_type, domain, results[15], ... }    │
└──────────────────────┬──────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────────┐
│ 阶段 3: 上下文构建                                               │
│                                                                  │
│ ┌─ build_context_with_file_injection() ─────────────────────┐  │
│ │  a. FileRegistry.detect_files_in_query()                   │  │
│ │     - 检测文件名引用 ("GB50052-2009", "闽电发展〔2015〕...") │  │
│ │     - 双向匹配: query-in-filename + filename-in-query      │  │
│ │                                                             │  │
│ │  b. 若命中 → get_full_document() 拉取完整文档               │  │
│ │     - 按 page_num 排序所有 chunks                          │  │
│ │     - 拼接为完整文档文本                                    │  │
│ │     - 预算检查: >10000 字符自动截断 (防 LLM 窗口溢出)       │  │
│ │     - 注入提示: "用户要查询的是文件 XXX 的完整内容"          │  │
│ │                                                             │  │
│ │  c. 若未命中 → 普通上下文                                    │  │
│ │     - format_context_for_llm() → 去重/格式化为 LLM prompt  │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                                                                  │
│ 返回第二个 SSE 心跳:                                              │
│   data: {"status": "thinking", "done": false,                    │
│          "sources_count": 5}                                     │
└──────────────────────┬──────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────────┐
│ 阶段 4: Prompt 构建 + 多轮对话上下文                             │
```

## 5. 文件入库完整流程

### 5.1 上传入库 (`POST /upload`)

```
前端选择文件 → Gradio upload_and_index()
  │
  ▼
POST /upload (multipart/form-data)
  Body:
    file: <binary>
    domain: "配电"          (可选, Form)
    category: "标准规范"     (可选, Form)
  │
  ▼
FastAPI upload_file():
  1. 校验文件类型 (.pdf/.doc/.docx/.xls/.xlsx/.ppt/.pptx/.txt/.md/.ofd)
  2. 保存到 data/uploads/ (重名自动加时间戳)
  3. FileProcessor.process(dest_path, domain, category)
     │
     ┌─ Step 1: _build_file_meta()
     │   - FileWalker 提取路径/文件名元数据
     │   - 计算文件 hash (MD5)
     │   - 检查 SQLite 注册表: 已存在且 completed → 跳过
     │
     ├─ Step 2: _parse_file()
     │   - PDF: PyMuPDF 文本提取 (前 15 页采样判断是否扫描件)
     │   - 扫描件: OCR Engine (PaddleOCR 子进程隔离)
     │   - DOC/DOCX: python-docx
     │   - XLS/XLSX: openpyxl
     │   - PPT/PPTX: python-pptx
     │   - OFD: 自定义 zipfile 提取器 (降级 ofdparser)
     │   - 返回: extracted_text, page_count, metadata
     │
     ├─ Step 3: Chunker.chunk()
     │   - 策略选择:
     │     · full_document (≤1500 字符) → 整文档 1 个 chunk
     │     · single_page (≤1 页) → 每页 1 个 chunk
     │     · semantic (其他) → 递归分割 512 token/chunk, overlap 64
     │
     ├─ Step 4: Embedder.embed_chunks()
     │   - BGE-M3 batch encode (batch_size=32)
     │   - 生成 dense (1024-dim) + sparse 向量
     │   - GPU 管理: 嵌入前确认 BGE-M3 已加载, OCR 已完成并卸载
     │
     ├─ Step 5: MilvusStore.insert()
     │   - 批量插入 chunk 向量 + 元数据
     │   - Collection: power_design_chunks (15 个元数据字段)
     │
     └─ Step 6: 更新 SQLite file_registry (status=completed)
  │
  ▼
返回 UploadResponse:
  {
    "success": true,
    "file_name": "GB50060-2008.pdf",
    "file_hash": "a1b2c3...",
    "status": "completed",
    "chunks_created": 45,
    "chars_extracted": 123456,
    "domain": "变电",
    "category": "标准规范",
    "doc_number": "GB50060-2008",
    "parse_time_ms": 2340.5,
    "embed_time_ms": 1520.3,
    "total_time_ms": 3860.8
  }
```

### 5.2 批量上传 (`POST /upload/batch`)

```
前端选择多个文件 → Gradio upload_and_index() 循环逐文件调用 POST /upload
  (非并行 — 避免 GPU 显存竞争)

或者:
POST /upload/batch (multipart/form-data, 多文件)
  → FastAPI 一次性接收所有文件 → FileProcessor.process_batch()
  → 顺序处理每个文件 → 返回 BatchUploadResponse
```

### 5.3 服务端路径入库 (`POST /upload/from-paths`)

```
无需上传, 直接指定服务端已有文件路径:
POST /upload/from-paths
  Body: {"paths": ["D:/docs/GB50060.pdf", ...], "domain": "变电"}
  → FileProcessor.process_batch() 顺序处理
  → 返回 BatchUploadResponse
```

### 5.4 崩溃恢复 (`POST /files/recover-pending`)

```
系统崩溃/进程被杀 → 部分文件状态为 "processing" 或 "pending"

恢复流程 (SSE 流式):
  1. _recover_stuck_files() → 将 "processing" 重置为 "pending"
  2. 查询所有 pending 文件
  3. 对每个文件重新执行 process()
  4. SSE 实时推送每个文件的恢复进度

前端 auto_recover_pending():
  - 页面加载时自动触发
  - 实时显示恢复进度表格
```

## 6. 多轮对话流程

### 6.1 会话生命周期

```
创建会话:
  POST /conversations {"title": "变电消防咨询"}
  → ConversationManager.create_conversation()
  → 返回: {"conv_id": "uuid-xxx", "title": "...", "created_at": "...", ...}

获取会话列表:
  GET /conversations
  → ConversationManager.list_conversations()
  → 返回: [{conv_id, title, preview, message_count, updated_at}, ...]

查看会话历史:
  GET /conversations/{conv_id}
  → ConversationManager.get_conversation(conv_id)
  → 返回: {conv_id, messages: [{role, content, timestamp, citations}], ...}

删除会话:
  DELETE /conversations/{conv_id}
  → ConversationManager.delete_conversation(conv_id)
  → 返回: {"status": "deleted"}
```

### 6.2 上下文压缩策略

```
ConversationManager 自动管理 token 预算:

max_context_tokens = 32768 (config.yaml: conversation.max_context_tokens)
compress_threshold = 0.85    (超过 85% 触发压缩)
keep_detail_rounds = 3       (保留最近 3 轮完整消息)

当估算 token 数 > 32768 * 0.85 = 27852 时:
  1. 保留最近 3 轮完整消息 (user + assistant)
  2. 更早的消息 → 摘要压缩
     "之前的对话摘要: 用户询问了... 助手回答了..."
  3. 将摘要作为 system message 的一部分

Token 估算公式:
  Chinese: chars / 1.5 ≈ tokens
  English: chars / 4.0 ≈ tokens
```

### 6.3 多轮对话的 API 交互

```
第 1 轮:
  POST /ask/stream {"query": "变电消防要求有哪些？", "conversation_id": null}
  → 后端创建 conv_id → 返回流式答案 → 前端记住 conv_id

第 2 轮 (使用同一个 conv_id):
  POST /ask/stream {"query": "那配电的呢？", "conversation_id": "uuid-xxx"}
  → ConvMgr 加载历史 → messages = [system, history..., "那配电的呢？" + 检索上下文]
  → LLM 在历史上下文中理解 "那...呢？" 指代的是消防要求 → 返回配电消防答案

第 3 轮:
  POST /ask/stream {"query": "对比一下两者的区别", "conversation_id": "uuid-xxx"}
  → ConvMgr 加载前两轮历史 → LLM 知道 "两者" = 变电消防 vs 配电消防
```

## 10. 错误处理机制

### 10.1 前端错误处理

```python
def _api(method, path, **kwargs):
    try:
        resp = requests.request(method, url, timeout=timeout, **kwargs)
        if resp.status_code >= 400:
            return f"[ERROR] {resp.status_code}: {resp.text[:200]}"
        return resp.json()
    except requests.ConnectionError:
        return f"[ERROR] 无法连接后端 ({API_BASE})，请先启动 API 服务"
    except requests.Timeout:
        return f"[ERROR] API 请求超时 ({timeout}s)"
    except Exception as e:
        return f"[ERROR] 请求失败: {e}"
```

### 10.2 后端降级策略

| 场景 | 降级行为 |
|------|---------|
| LLM 未部署 (Ollama 未启动) | 返回检索结果原文 + 警告 "⚠ LLM 未部署" |
| LLM 生成异常 | 返回 "⚠ LLM 不可用: {error}" + 检索到的资料列表 |
| Milvus 检索无结果 | 尝试 FileRegistry 精确匹配 → 无则返回 "未找到相关内容" |
| 嵌入模型加载失败 | 启动时 catch 异常, 打印警告 (不阻止服务启动) |
| OCR 子进程崩溃 | 逐个页面重试, 全部失败则标记文件状态为 "failed" |
| GPU 显存不足 | `wait_for_vram()` 阻塞等待, 最多等 120s |

### 10.3 文件入库状态机

```
                    ┌──────────┐
     新文件 ──────→ │ pending  │
                    └────┬─────┘
                         │ process() 开始
                    ┌────┴──────┐
                    │ processing│
                    └────┬──────┘
                    ┌────┴──────────────────┐
                    │                       │
              ┌─────┴─────┐           ┌────┴─────┐
              │ completed │           │  failed   │
              └───────────┘           └──────────┘
                    │                       │
              reindex()               reindex()
              (删除→重建)              (删除→重建)

崩溃恢复:
  processing (残留) → _recover_stuck_files() → pending → process()
  pending → POST /files/recover-pending → completed | failed
```

### 10.4 关键超时配置

| 位置 | 超时 | 说明 |
|------|------|------|
| Gradio → API | 120s 连接, 900s 读取 | 应对大文件 OCR (单文件可达 10min) |
| API → LLM (Ollama) | 无显式超时 | 依赖 SSE 流式, 由 Ollama 进程管理 |
| Milvus 操作 | 默认 | Milvus Lite 本地文件, 无网络开销 |
| OCR 子进程 | 600s/页 | communicate() timeout, 超时则 kill |
| GPU 等显存 | 120s | wait_for_vram() 最大等待时间 |
