## 2. 完整入库流程

```
用户输入 (API / UI / CLI)
  │
  ▼
FileProcessor.process(file_path)
  │
  ├── [1] SHA256 去重检查 ── SQLite file_registry
  │
  ├── [2] 构建文件元数据 ── FileWalker.extract_path_metadata() + extract_filename_metadata()
  │                         提取: domain, category, doc_number, voltage_level, discipline, ...
  │
  ├── [3] 解析文件 ─── 按扩展名路由到 _parse_file():
  │   │
  │   ├── PDF + OCR 渐进路径 → _process_pdf_progressive()
  │   │   ├── PDFParser.parse() ── fitz 文本提取 (CPU)
  │   │   │   └── 两阶段采样: 前15页判断文档类型 (扫描件/文字型)
  │   │   │   └── 每页 get_text() 有 try/except 容错, 坏页不影响整文档
  │   │   ├── 非 OCR 页面文本 → Chunker.chunk_text_document()
  │   │   ├── ★ 卸载 BGE-M3 ── embedder.unload() → 释放 ~2GB 显存
  │   │   ├── ★ OCR 子进程 (_ocr_pdf_subprocess):
  │   │   │   ├── ThreadPool(max=8) 渲染 PDF 页面为 PNG
  │   │   │   │   └── _render_page_to_png(): max_dim=3000 超限 LANCZOS 缩放
  │   │   │   │       (9744×6890 → 3000×2121, OCR 从 660s 降至 2.6s/页)
  │   │   │   └── 单子进程 PaddleOCR 批量推理
  │   │   │       └── _spawn_ocr_worker(): subprocess.Popen + communicate()
  │   │   │       └── 超时后 cmd //c taskkill /F /T /PID 杀进程树
  │   │   ├── ★ 重新加载 BGE-M3 ── embedder.reload()
  │   │   └── OCR 文本分块 → Chunker
  │   │
  │   └── 其他格式 → _parse_file() (详见第5节格式支持)
  │
  ├── [4] 嵌入向量 (GPU)
  │   ├── Embedder.encode() ── BGE-M3 dense(1024维) + sparse 向量
  │   ├── create_text_for_embedding(): "[元数据字段] 正文内容"
  │   └── batch_size=32, 串行批处理
  │
  ├── [5] 写入向量库
  │   ├── MilvusStore.delete_by_file_hash() ── 清理旧索引 (按 chunk_id LIKE prefix)
  │   └── MilvusStore.insert(batch_size=500) ── 批量写入 Milvus Lite
  │
  └── [6] 更新 SQLite 注册表 ── file_registry (status/chunks/chars/time/error)
```

### 2.1 非 PDF 文件的简化路径

```
文件 (.docx/.xlsx/.txt/...)
  → _parse_file() → 解析器提取文本 → Chunker 分块
  → _embed_and_insert() → BGE-M3 嵌入 → Milvus 写入
  (无 OCR, 无 BGE-M3 卸载/重载)
```

## 3. 并发模型

### 3.1 进程架构

```
┌─────────────────────────────────────────────────────┐
│  主进程 (anaconda python.exe)                        │
│  ├── FastAPI (uvicorn, 多 worker 线程)              │
│  ├── BGE-M3 嵌入模型 (SentenceTransformer, CUDA)     │
│  ├── BGE-Reranker (FlagEmbedding, CUDA, 按需)       │
│  ├── PyMuPDF 渲染 (CPU, ThreadPoolExecutor)          │
│  └── SQLite + Milvus Lite (进程内)                  │
│                                                      │
│  ┌─ OCR 子进程 (PPOCRLabel venv python.exe) ─┐       │
│  │  PaddleOCR GPU (paddlepaddle-gpu 3.2.2)    │       │
│  │  独立 CUDA Context                         │       │
│  └────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────┘

  Ollama 服务 (独立进程, ollama.exe)
  └── qwen3.5:4b (~4GB 显存)
```

### 3.2 并行/串行矩阵

| 操作 | 执行模型 | 并行度 | 锁/约束 |
|------|----------|--------|---------|
| `process_batch()` | **串行** | 1 文件/次 | 逐个调用 `process()` |
| `process()` 管道 | **串行** | 解析→分块→嵌入→写入 | 阶段间顺序依赖 |
| PDF 文本提取 (fitz) | 串行 | 逐页 | - |
| PNG 渲染 (PyMuPDF) | **ThreadPoolExecutor** | min(8, max(4, total)) | - |
| PaddleOCR 推理 | **子进程** Popen+communicate | **1 子进程** | `_ocr_lock` 全局互斥 |
| OCR turbo 模式 | ThreadPool spawn N×Popen | ≤2 子进程 (VRAM 感知) | `_ocr_lock` 全局互斥 |
| 嵌入 (BGE-M3) | 串行 batch 循环 | batch_size=32 | - |
| Milvus 写入 | 串行 batch 循环 | batch_size=500 | - |
| SQLite 注册表 | 串行, autocommit | - | - |
| GPU 显存背压 | 轮询循环 | - | `wait_for_vram()` |

### 3.3 三个关键约束

**约束 1: `_ocr_lock` (全局互斥锁)**

`threading.Lock`, 位于 `ocr_engine.py` 第 41 行。`ocr_pdf_pages()` 和 `ocr_page()` 都通过 `with _ocr_lock:` 获取。同一时刻最多一个 OCR 子进程在运行。防止并发 API 请求导致 2+ 个 PaddleOCR 实例同时占用 GPU。

**约束 2: OCR 子进程使用独立 Python 环境**

GPU 模式下使用 `PPOCRLabel\.venv\Scripts\python.exe` (paddlepaddle-gpu 3.2.2)，与主进程 (anaconda, pymilvus protobuf≥5.27.2) 隔离 protobuf 版本冲突。环境变量 `PROTOCOL_BUFFERS_PYTHON_IMPLEMENTATION=python` 绕过 C++ protobuf 实现。

**约束 3: communicate() 模式替代 --stream**

使用 `subprocess.Popen` + `proc.communicate(input=json, timeout=N)` 替代脆弱的 `--stream` 管道模式。`communicate()` 内部用线程并发读取 stdout/stderr，无死锁风险。超时后 `cmd //c taskkill /F /T /PID` 杀整个进程树 (含 PaddlePaddle CUDA 孙进程)。

## 4. 资源占用估算

### 4.1 GPU 显存 (RTX 4070 SUPER 12GB)

| 模型 | 估算占用 | 加载时机 | 生命周期 |
|------|----------|----------|----------|
| BGE-M3 (嵌入) | ~2.0 GB | 首次嵌入时延迟加载 | 常驻 (OCR 前卸载, OCR 后重载) |
| BGE-Reranker v2-m3 | ~2.0 GB | 检索时按需加载 | 按需加载/卸载 |
| PaddleOCR (GPU) | ~1.5-3.0 GB | OCR 子进程启动时 | 瞬时 (子进程退出即释放) |
| Ollama (qwen3.5:4b) | ~4.0 GB | Ollama 服务管理 | 独立进程, `keep_alive` 控制 |
| WDDM 系统预留 | ~1.7 GB | 始终 | 始终 |

**OCR 阶段的显存编排流程**:

```
阶段                     显存分布                        空闲
─────────────────────────────────────────────────────────────
空闲态:   BGE-M3(2G) + WDDM(1.7G) = 3.7G             ~8.3G
OCR准备:  卸载BGE→torch.cuda.empty_cache()            ~10.3G
OCR运行:  WDDM(1.7G) + PaddleOCR(1.5-3G) = ~4.7G     ~7.3G
OCR完成:  子进程退出→WDDM回收                          ~10.3G
重载BGE:  BGE-M3(2G) + WDDM(1.7G) = 3.7G             ~8.3G
```

**Ollama 共存场景** (LLM+嵌入+OCR 全在线):
```
BGE-M3(2G) + Ollama(4G) + Reranker(2G) + WDDM(1.7G) = 9.7G
空闲仅 ~2.3G — 不够 PaddleOCR(1.5-3G)
→ _free_gpu_for_ocr() 通知 Ollama 卸载模型 (keep_alive=0)
→ 释放 4G → 空闲 ~6.3G → 够 PaddleOCR
```

### 4.2 磁盘

| 项目 | 估算 |
|------|------|
| OCR 临时 PNG (dpi=150, max_dim=3000) | ~1-2 MB/页, 批处理后即时删除 |
| Milvus Lite DB | ~数百 MB (随 chunks 增长) |
| SQLite file_metadata.db | ~130 KB |
| 上传文件暂存 (data/uploads) | 按文件大小, 通常 1-60 MB/文件 |

### 4.3 内存 (RAM)

| 组件 | 估算 |
|------|------|
| FastAPI + uvicorn | ~100 MB |
| BGE-M3 模型权重 | 仅 GPU (~2.2 GB), 无显著 RAM 副本 |
| PyMuPDF 渲染缓冲 | 按需, ~数十 MB |
| Chunk 对象列表 | 总字符数 × 2 overhead |
| Python 进程堆 | ~200-500 MB |

## 5. 格式支持

| 格式 | 扩展名 | 解析器 | 解析策略 | 分块策略 |
|------|--------|--------|----------|----------|
| PDF (文档) | `.pdf` | PyMuPDF (fitz) + PaddleOCR | 两阶段采样, 文本提取/OCR 回填 | semantic / single_page / full_document |
| PDF (图纸) | `.pdf` | PyMuPDF 单页快速 | 单页全文本提取 | 1 chunk |
| Word 新版 | `.docx` | python-docx | 段落 + 表格 (含 pipe 分隔) | semantic |
| Word 旧版 | `.doc` | win32com → LibreOffice → olefile → antiword → docx2txt | 6 级回退链 | semantic |
| Excel 新版 | `.xlsx` | openpyxl (data_only=True) | 按 Sheet → 行 → 单元格 | semantic |
| Excel 旧版 | `.xls` | xlrd | 按 Sheet → 行 → 单元格 | semantic |
| PPT 新版 | `.pptx` | python-pptx | 按 Slide → Shape → 文本 | semantic |
| PPT 旧版 | `.ppt` | LibreOffice → PowerPoint COM | 2 级回退 | semantic |
| WPS 文字 | `.wps` | python-docx → zipfile XML → LibreOffice → olefile → WPS COM → Word COM | 6 级回退链 | semantic |
| 纯文本 | `.txt`, `.md` | `open()` UTF-8 | 直接读取 | semantic |
| 国产版式 | `.ofd` | ofdparser | 按页 → 元素 | semantic |
| CAD 图纸 | `.dwg`, `.dxf` | **跳过** | 需专用解析器 | - |

### 5.1 解析器回退链说明

对于 `.doc` 和 `.wps` 这类遗留格式, 系统按优先级依次尝试多种解析后端:

```
.doc 回退链: win32com (MS Word COM) → LibreOffice headless → olefile raw → antiword (Linux) → docx2txt → python-docx
.wps 回退链: python-docx → zipfile XML → LibreOffice headless → olefile raw → WPS COM → MS Word COM
```

任一后端成功解析即返回, 全部失败则打印警告并返回空列表。

## 6. 核心类与方法

### 6.1 FileProcessor (`src/ingestion/file_processor.py`)

| 方法 | 签名 | 说明 |
|------|------|------|
| `process` | `(file_path, domain, category, progress_callback, turbo, turbo_max_workers) → ProcessResult` | **主入口**, 完整管道: 解析→分块→嵌入→写入 |
| `process_batch` | `(file_paths, ...) → BatchResult` | **串行**逐个调用 `process()` |
| `_process_pdf_progressive` | `(...) → ProcessResult` | PDF+OCR 专用: 解析→卸载BGE→OCR子进程→重载BGE→嵌入→写入 |
| `_embed_and_insert` | `(chunks, ...) → ProcessResult` | 嵌入 BGE-M3 + 写入 Milvus |
| `_parse_file` | `(file_path, file_meta) → List[Chunk]` | 按扩展名路由到具体解析器 |
| `delete` | `(identifier, remove_file) → bool` | 从 Milvus + 注册表删除 |
| `reindex` | `(identifier, progress_callback) → ProcessResult` | 先 delete 再 process |
| `sync_orphans` | `(dry_run) → dict` | 扫描并清理注册表中的孤记录 |
| `_build_file_meta` | `(file_path, file_hash, domain, category) → dict` | 构建元数据字典 (调用 FileWalker) |

### 6.2 OCREngine (`src/ingestion/ocr_engine.py`)

| 方法 | 说明 |

| `_get_ocr_python() → str` | GPU 模式返回 PPOCRLabel venv Python 路径 |

### 6.5 Chunker (`src/ingestion/chunker.py`)

| 策略 | 触发条件 | 方法 | 说明 |
|------|----------|------|------|
| `full_document` | chars < 1500 | `_make_full_document_chunk()` | 整文档不分块 |
| `single_page` | 1 页 PDF | `_make_per_page_chunks()` | 一页一块 |
| `semantic` | 多页/长文本 | `_make_semantic_chunks()` | `_recursive_split()` 中文优先分隔符 |

递归分隔符优先级: `\n\n\n` → `\n\n` → `\n` → `。` → `；` → `，` → `. ` → ` ` → 字符级强拆

Chunk 大小: `chunk_size=512` tokens × 3 ≈ 1536 中文字符, `chunk_overlap=64` tokens

### 6.6 Embedder (`src/ingestion/embedder.py`)

|------|------|

| `unload()` | `del model` → `gc.collect()` → `torch.cuda.empty_cache()` |

### 6.7 MilvusStore (`src/ingestion/milvus_store.py`)

| 方法 | 说明 |
|------|------|
| `insert(chunks, dense_vectors, sparse_vectors, embedding_texts, batch_size=500)` | 批量写入 Milvus Lite |
| `delete_by_file_hash(hash)` | `chunk_id LIKE "{hash}%"` 前缀删除 |
| `hybrid_search(dense, sparse, filter, limit, rrf_k, dense_weight, sparse_weight) → List[dict]` | RRF 混合检索 |
| `query_by_file_hash(hash) → List[dict]` | 按文件哈希查询所有 chunks |

Schema: 31 字段, 主键 `chunk_id` (VARCHAR 256), 向量字段 `dense_vector` (FLOAT_VECTOR 1024), `sparse_vector` (SPARSE_FLOAT_VECTOR). Collection 名: `power_design_chunks`.

### 6.8 GpuMonitor (`src/utils/gpu_monitor.py`)

| 方法 | 说明 |
|------|------|
| `get_vram_info() → dict` | 返回 `{total_mb, used_mb, free_mb, effective_free_mb, torch_reserved_mb, wddm_cached_mb}` |
| `wait_for_vram(min_free_mb, timeout_s) → bool` | **LLM 优先背压**: Ollama 繁忙→等待; 空闲但显存不够→30s 后强制执行 |
| `is_ollama_busy() → bool` | 检查 Ollama `/api/ps` 是否加载了模型 |
| `get_ocr_workers_possible(vram_per_worker_mb, safety_margin_mb) → int` | 计算安全并行 OCR 子进程数 (上限 3) |
| `log_vram_snapshot(tag)` | 打印 GPU + 系统 RAM 快照 |

`effective_free_mb` 计算逻辑: `nvidia-smi free + max(0, nvidia-smi used - torch_reserved - 1700_system_overhead)` — 因为 WDDM 缓存可被新 CUDA context 回收复用。

## 7. 配置关键项

```yaml
# ===== OCR 配置 =====
ocr:
  enabled: true                    # 是否启用扫描件 OCR
  dpi: 150                         # PDF 渲染分辨率
  max_image_dim: 3000              # ★ 图片最大边长, 超限 LANCZOS 缩放

  use_gpu: true                    # GPU 推理
  use_subprocess: true             # 子进程隔离 (protobuf 冲突)
  cpu_threads: 4                   # GPU 模式下自动忽略
  ocr_tmp_dir: "E:/RongNengRAG/data/ocr_tmp"  # 避开系统盘

  # GPU 显存管理
  gpu_memory:
    min_free_vram_mb: 2500         # 入库前最少空闲显存
    poll_interval_s: 5
    max_wait_s: 300                # 最长等待, 超时强制执行

  # OCR 进程池 (turbo, 默认关闭)
  turbo:
    enabled: false                 # 大文件可开启, ≤2x 子进程
    max_workers: 0                 # 0=自动(VRAM感知)
    pages_per_worker: 25           # 每子进程处理页数

# ===== 分块配置 =====
chunking:
  semantic:
    chunk_size: 512                # tokens
    chunk_overlap: 64              # tokens
  single_page:
    enabled: true
    max_pages_for_single: 1
  full_document:
    max_chars_for_full: 1500

# ===== 嵌入配置 =====
embedding:
  provider: sentence_transformers
  model_name: BAAI/bge-m3
  device: cuda
  batch_size: 32

# ===== 检索配置 =====
retrieval:
  coarse_top_k: 50                 # 粗排候选数
  fine_top_k: 15                   # 精排后返回数
  rrf_k: 60                        # RRF 融合参数
  dense_weight: 0.5                # 稠密权重
  sparse_weight: 0.5               # 稀疏权重
```

## 8. 常见问题排查

### 8.1 入库超时

**症状**: 文件长时间卡在 `processing` 状态, 最终超时失败

**常见原因**:

| 原因 | 诊断方法 | 解决 |
|------|----------|------|
| 超大工程图 (65"×46") dpi=150 渲染出 9744×6890px | 检查 `ocr_tmp` 中 PNG 尺寸 | `max_image_dim: 3000` 自动缩放 |
| 两个 OCR 子进程同时运行 | `wmic process where "name='python.exe'" get commandline` | 杀孤儿进程, `_ocr_lock` 保底 |
| Ollama 占用全部显存 | `nvidia-smi` 查看显存分布 | `_free_gpu_for_ocr()` 通知 Ollama 卸载 |
| PDF 单页损坏导致整文档失败 | 注册表 error 含 `code=7: object is not a stream` | 已修复: `parse()` 逐页 try/except 容错 |

### 8.2 显存溢出 (OOM)

**症状**: `torch.cuda.OutOfMemoryError` 或系统卡死

**排查步骤**:
1. `nvidia-smi` 查看当前显存使用
2. `wmic process where "name='python.exe'" get commandline` 检查是否有孤儿 OCR 进程
3. 检查 Ollama 是否加载了大模型 (`curl http://localhost:11434/api/ps`)
4. 杀孤儿进程: `taskkill /F /PID <pid>`

### 8.3 解析后无有效文本

**症状**: `status=failed`, `error_message="解析后无有效文本内容"`

**原因**: 文件是纯图片扫描件且 OCR 未识别到文本, 或文件格式不支持 (如 .dwg)

**解决**: 对于扫描件, 确认 `ocr.enabled: true`; 对于 .dwg/.dxf, 需专用解析器 (当前不支持)
