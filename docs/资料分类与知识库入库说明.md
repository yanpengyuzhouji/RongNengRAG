## 2. 支持的文件格式

### 2.1 格式—解析能力矩阵

| 格式 | 扩展名 | 解析方式 | 全扫描件OCR | 备注 |
|------|--------|----------|-------------|------|
| **PDF（文字型）** | `.pdf` | PyMuPDF 逐页文本提取 | 自动检测 → PaddleOCR | 首选格式 |
| **PDF（扫描件）** | `.pdf` | 采样15页检测 → 全扫描跳过逐页 → 页面渲染+OCR | ✅ 138页/4.3分钟 | 速度 = 页数 × 1.9s |
| **Word 新版** | `.docx` | python-docx 段落提取 | — | 推荐 |
| **Word 旧版** | `.doc` | Word COM → LibreOffice → olefile → antiword（5级回退） | — | 推荐转存 .docx |
| **WPS 文字** | `.wps` | python-docx → zipfile → LibreOffice → olefile → WPS COM（6级回退） | — | 新版 .wps 本质 .docx |
| **Excel** | `.xls/.xlsx` | openpyxl 逐Sheet读取 | — | 表格合并为文本 |
| **纯文本** | `.txt/.md` | 直接 UTF-8 读取 | — | — |
| **OFD** | `.ofd` | 自定义提取器 (标准库) + ofdparser 回退 | — | 国产版式文档，支持 GBK/Unicode/CMap |
| **PPT** | `.ppt/.pptx` | 上传白名单支持 | — | 入口支持，解析待强化 |
| **CAD图纸** | `.dwg/.dxf` | 跳过（需专用解析器） | — | 暂不支持内容解析 |

### 2.2 分块策略（自动选择）

| 策略 | 触发条件 | 行为 |
|------|----------|------|
| 全文分块 | 全文 < 1,500 字符 | 整个文档不拆分，单个 chunk |
| 单页分块 | PDF ≤ 1 页 | 每页 = 1 个 chunk |
| 语义分块 | 其他所有情况 | 512 token 窗口 + 64 token 重叠，按段落/句号切分 |

## 3. 文件名规范（强烈建议）

### 3.1 标准编号类

遵循 `标准编号_中文名称_附加信息.pdf` 格式：

```
✅ GB_50060-2008_3~110kV高压配电装置设计规范_OCR.pdf
✅ GB_50169-2016_电气装置安装工程_接地装置施工及验收规范_OCR.pdf
✅ DL_5053-2012_火力发电厂职业安全卫生设计规程.pdf
✅ QGDW_10173-2023_变电站模块化建设典型方案.pdf

❌ 高压配电装置设计规范.pdf          # 缺少标准编号，无法自动提取编号
❌ GB50060.pdf                       # 缺少中文名，检索匹配率低
```

### 3.2 会议材料类

遵循 `序号+材料名称_时间戳.pdf` 格式：

```
✅ 01会议材料之一2022年公司设计管理情况通报.pdf
✅ 02会议材料之二基建设计管理文件宣贯.pdf
✅ 03会议材料之三2023年设计管理重点工作.pdf
```

### 3.3 公司发文类

```
✅ 闽电发展〔2015〕241号_关于加强变电消防管理的通知.pdf
✅ 榕能基建〔2022〕18号_关于配电自动化改造的通知.pdf
```

### 3.4 扫描件标识

扫描OCR处理的文件，建议在文件名末尾加 `_OCR`：

```
✅ GB_50086-2015_岩土锚杆与喷射混凝土支护工程技术规范_OCR.pdf
✅ GB_50191-2012_构筑物抗震设计规范_OCR.pdf
```

> 加 `_OCR` 仅便于人工识别扫描件来源，**不影响系统自动检测**——系统通过采样前15页文字量自动判断是否需要 OCR。

## 4. 自动元数据提取规则

系统从目录路径和文件名中自动提取以下标签：

| 元数据字段 | 提取来源 | 示例 |
|-----------|----------|------|
| `domain` | 目录路径第1段 | `变电/标准规范/xxx.pdf` → 变电 |
| `category` | 目录路径第2段 | `变电/标准规范/xxx.pdf` → 标准规范 |
| `doc_number` | 文件名正则匹配 | `GB 50060-2008` → `doc_number: GB 50060-2008` |
| `publish_level` | 文件名关键词 | 含"GB" → 国标；含"闽电" → 省公司_福建 |
| `voltage_level` | 文件名正则匹配 | 含"500kV" → 500kV 电压等级 |
| `year` | 文件名4位数字 | `GB 50060-2008` → 2008 |
| `discipline` | 文件名关键词 | 含"继电保护" → 保护；含"通信" → 通信 |
| `equipment_type` | 文件名关键词 | 含"变压器" → 变压器；含"GIS" → 开关柜GIS |
| `region` | 文件名关键词 | 含"福建/闽/福州/榕" → 福建/福州 |

## 5. 入库流程与性能基准

### 5.1 入库管道

```
文件上传
  ├─ 1. SHA-256 哈希 → 已入库跳过
  ├─ 2. 元数据提取（路径 + 文件名正则）
  ├─ 3. 解析
  │   ├─ PDF: 采样15页判断类型 → 文字型逐页/扫描件全OCR
  │   ├─ DOC/DOCX/WPS: Word文档解析
  │   └─ XLSX/TXT/MD: 直接读取
  ├─ 4. 分块 (Chunker)
  ├─ 5. 向量嵌入 (BGE-M3 GPU)
  └─ 6. 写入 Milvus 向量库
```

### 5.2 性能基准（RTX 4070 SUPER 12GB）

| 文件类型 | 页/大小 | 解析耗时 | OCR耗时 | 嵌入耗时 | 总耗时 |
|----------|---------|----------|---------|----------|------|
| 文字型PDF（小） | 59页 / 1.5MB | ~6s | 3页 × 5s | ~13s | ~34s |
| 文字型PDF（中） | 208页 / 1.3MB | <1s | 4页 × 5s | ~19s | ~39s |
| 文字型PDF（大） | 453页 / 13.4MB | ~9s | 4页 × 5s | ~64s | ~93s |
| 扫描件PDF | 138页 / 7.4MB | <1s | 138页 × 5.7s = 787s | ~28s | ~815s (~14分钟) |
| 超大工程图扫描件 | 1页 / 65"×46" | — | 2.6s (缩放至3000px) | ~1s | ~4s |
| DOCX | 不限 | ~2s | — | 按chunk数 | ~5-10s |
| DOC | 不限 | ~5-20s | — | 按chunk数 | ~10-30s |

> **结论**：文字型 PDF 可以秒级入库；全扫描件 PDF 的速度瓶颈在 OCR（GPU ~5-7s/页），138 页约 14 分钟。超大工程图通过 max_image_dim=3000 自动缩放后从 660s 降至 2.6s/页。

### 5.3 入库方式

| 方式 | 命令/操作 | 适用场景 |
|------|----------|----------|
| **API 上传** | `POST /upload` (multipart) | 单个/少量文件，web UI |
| **API 路径** | `POST /upload/from-paths` (JSON列表) | 服务端已有文件，批量路径指定 |
| **UI 上传** | Gradio Tab "文件入库"，可多选 | 日常使用 |
| **直接调用** | `processor.process(path, domain, category)` | 脚本批量处理 |

## 8. 批量入库脚本模板

### 8.1 命令行模式（服务端已存文件）

```python
# batch_ingest.py — 批量入库脚本
import requests, json, os

API = "http://localhost:8000"
KB = r"E:\RongNengRAG\data\uploads"
DOMAIN = "送电输电"
CATEGORY = "标准规范"

# 收集所有 PDF
pdfs = []
for root, dirs, files in os.walk(KB):
    for f in files:
        if f.lower().endswith('.pdf'):
            pdfs.append(os.path.join(root, f))

print(f"共 {len(pdfs)} 个 PDF 待入库")

# 分批发送（每次最多 5 个，避免单次超时）
batch_size = 5
for i in range(0, len(pdfs), batch_size):
    batch = pdfs[i:i+batch_size]
    resp = requests.post(
        f"{API}/upload/from-paths",
        json=batch,
        params={"domain": DOMAIN, "category": CATEGORY},
        timeout=600
    )
    result = resp.json()
    print(f"[{i//batch_size+1}/{(len(pdfs)-1)//batch_size+1}] "
          f"成功={result['success_count']}, 失败={result['failed_count']}")
    for r in result['results']:
        if not r['success']:
            print(f"  FAIL: {r['file_name']} — {r.get('error_message','?')}")
```

### 8.2 直接调用模式（服务关闭时，更高性能）

```python
# direct_batch_ingest.py — 直接调用 FileProcessor
import sys; sys.path.insert(0, 'src')
from ingestion.file_processor import FileProcessor
import os, glob

processor = FileProcessor()
files = glob.glob(r"E:\RongNengRAG\data\uploads\*.pdf")

for fp in files:
    print(f"入库: {os.path.basename(fp)}")
    r = processor.process(fp, domain="变电", category="标准规范")
    print(f"  -> {r.status.value}, chunks={r.chunks_created}, "
          f"chars={r.chars_extracted:,}, {r.total_time_ms/1000:.1f}s")
```

> ⚠ 直接模式要求 API 服务未运行（Milvus 排他锁），适合大批量离线入库。

## 9. OCR 扫描件识别专项说明

### 9.1 工作原理

```
PDF 解析
  ├─ 阶段1: 采样前15页，fitz 提取文本
  │   ├─ >70% 页面无文字 → 标记全扫描件 (ocr_skip_fitz)
  │   └─ 否则 → 逐页 fitz，按需标记 needs_ocr
  └─ 阶段2: OCR 处理
      ├─ ★ 卸载 BGE-M3 (释放 ~2GB 显存)
      ├─ 子进程隔离 (PPOCRLabel venv, PROTOCOL_BUFFERS_PYTHON_IMPLEMENTATION=python)
      ├─ 单子进程 communicate() 批量推理 (无死锁风险)
      ├─ PaddleOCR PP-OCRv5_server 中文模型 (GPU, ~5-7s/页)
      ├─ 超大图自动缩放: max_image_dim=3000 LANCZOS (9744×6890→2.6s/页)
      ├─ 识别文本回填至解析结果
      └─ ★ 重载 BGE-M3 (恢复嵌入能力)
```

### 9.2 OCR 配置（config.yaml）

```yaml
ocr:
  enabled: true              # 总开关：设为 false 跳过所有 OCR
  provider: "paddleocr"      # 后端
  lang: "ch"                 # 中文
  dpi: 150                   # 页面渲染分辨率 (150 DPI 平衡速度与准确率)
  max_image_dim: 3000        # 超大图缩放阈值 (9744×6890→3000px, 254x 提速)
  min_text_chars: 50         # 低于此字符数的页面触发 OCR
  use_subprocess: true       # 子进程隔离 (protobuf 冲突解决)
  use_gpu: true              # GPU 推理 (~5-7s/页)
  ocr_tmp_dir: "E:/RongNengRAG/data/ocr_tmp"  # 避开系统盘
  gpu_memory:
    min_free_vram_mb: 2500   # 入库前最少空闲显存
    max_wait_s: 300          # 显存等待超时
  turbo:
    enabled: false           # VRAM感知多子进程并行 (大文件可选，默认关闭)
    max_workers: 0           # 0=VRAM自动, 1=安全单进程
```

### 9.3 入库耗时与 OCR 关系

> 核心原则：**入库耗时 ≈ 扫描件页数 × 5-7s (GPU) + 嵌入耗时**

非扫描件 PDF 即使数百页，入库也只需几十秒；只有扫描页才耗时。超大工程图通过 `max_image_dim=3000` 自动缩放，可从 660s/页 降至 2.6s/页。

## 10. 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 文件卡在 `processing` 状态 | 入库过程被中断（服务器重启等） | 删除该文件记录后重新 `reindex` |
| chunks=0 入库完成 | 文件解析后无有效文本（扫描件无文字+OCR关闭） | 开启 `ocr.enabled: true` |
| 入库后搜索不到 | Milvus 索引未刷新 | 确认 `collection_exists` 且 `load_collection` |
| DOC 文件 chunks=0 | 旧版 .doc 解析失败（无 Word COM 环境） | 先另存为 .docx 再入库 |
| OCR 速度慢 | GPU 推理已启用 (PPOCRLabel venv 子进程隔离) | GPU ~5-7s/页，超大图 max_image_dim=3000 可 254x 提速 |
| 文件名乱码显示 | 中文文件名编码问题 | 确保源文件名 UTF-8，入库 `_` 替代空格 |
